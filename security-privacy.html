Security & Privacy安全与隐私

A security tool that earns your trust — not asks for it. 可验证的信任，而非要你盲信。

Sieve sits in the most sensitive spot in your stack: between your AI agent and the model API, where every prompt and response flows through. So it holds itself to the strictest possible rule — detection runs 100% locally, and it never sees, stores, or sends a single line of your prompts, responses, API keys, or activity. Sieve 处在你技术栈里最敏感的位置：AI agent 与模型 API 之间，所有 prompt 和 response 都流经此处。正因如此它对自己施加最严格的约束——检测推理 100% 本地完成，永不查看、存储或发送你的任何 prompt、response、API key 或使用记录。

🔒 100% local detection检测 100% 本地 📡 Only 3 permitted outbound paths仅 3 类允许出站 🚫 No local CA · no MITM of your traffic不装本地 CA · 不 MITM
Trust boundary信任边界

Where your data lives — and the only places anything leaves. 你的数据待在哪里——以及任何东西离开机器的唯一出口。

Everything inside the dashed boundary stays on 127.0.0.1. Only three categories of traffic ever cross it — and none of them carries your prompts or secrets. 虚线边界内的一切都留在 127.0.0.1。只有三类流量会跨越这条边界——且没有一类携带你的 prompt 或密钥。

The privacy promise隐私承诺

What leaves your machine — exactly, and nothing more. 离开你机器的，仅此而已。

Telemetry beacon — 4×/day遥测信标 — 每天 4 次

Sieve checks for updates four times a day. That request carries five fields only — enough to count installs and ship the right update, nothing that identifies you: Sieve 每天 4 次检查更新。该请求仅附 5 个字段——刚好够统计装机量并下发正确更新，没有任何能识别你的信息：

  • 1version — the Sieve build you runversion — 你运行的 Sieve 版本
  • 2OS — e.g. macOSOS — 操作系统，如 macOS
  • 3CPU arch — arm64 / x86_64CPU arch — CPU 架构 arm64 / x86_64
  • 4install-id — a random UUID, not tied to any account or deviceinstall-id — 本地随机生成的 UUID，不绑定账号或设备
  • 5channel — stable / betachannel — 更新通道 stable / beta
Never uploaded — by design永不上传 — 设计如此

There is no code path in Sieve that transmits any of the following. Detection is local, so it simply never needs them to leave: Sieve 中不存在任何能发送下列内容的代码路径。检测在本地完成，所以它们根本不需要离开你的机器：

  • Your prompts — never你的 prompt — 永不
  • Model responses — never模型 response — 永不
  • Your API keys — never (they pass through to your chosen upstream only)你的 API key — 永不（仅透传给你指定的上游）
  • Any usage records — what you asked, what was blocked: never任何 使用记录 — 你问了什么、拦了什么：永不

Two switches, fully in your control两个开关，完全由你掌控

SIEVE_NO_TELEMETRY=1 Stops the install-count beacon. Rule updates still work — your protection stays current. 停止装机统计信标。规则更新照常工作——你的防护仍保持最新。
SIEVE_NO_UPDATE=1 Disables update checks entirely. No outbound connection to updates.sieveai.dev at all. 完全禁用更新检查。彻底不再连接 updates.sieveai.dev。
Allowed egress允许出站

The only three connections Sieve is allowed to make. Sieve 仅被允许发起的三类连接。

Anything outside this list is, by definition, not Sieve. Audit it yourself with a packet sniffer — you will see these three and nothing else. 列表之外的任何流量，按定义就不是 Sieve。你可以用抓包工具自行核验——只会看到这三类，别无其他。

No CA · No MITM不装 CA · 不做 MITM

Sieve does not install a root certificate or hijack your TLS. Sieve 不安装根证书，也不劫持你的 TLS。

Classic DLP and "inspection" proxies install a local certificate authority and man-in-the-middle all your encrypted traffic — they become a master key to everything you do. Sieve refuses that design. 传统 DLP 和"检查"代理会安装本地证书颁发机构（CA），对你所有加密流量做中间人——它们等于拿到了你一切活动的万能钥匙。Sieve 拒绝这种设计。

  • No system-wide root CA is installed — your other apps' TLS is untouched.不安装系统级根 CA——你其它应用的 TLS 毫发无损。
  • Sieve only handles the agent endpoint you point at it (127.0.0.1). It is a deliberate, scoped proxy — not a transparent interceptor.Sieve 只处理你显式指向它的 agent 端点（127.0.0.1）。它是有意为之、范围明确的代理——不是透明拦截器。
  • Remote-script bootstrap (curl | sh) is exactly the attack surface Sieve guards against — so we never ship one.远程脚本盲跑（curl | sh）正是 Sieve 要防的攻击面——所以我们绝不提供。
Fail-closedFail-closed

When in doubt, Sieve stops — it never waves things through. 遇到不确定，Sieve 会拦下——绝不放行。

A security control that fails open is theater. Sieve is built fail-closed: if a Critical tool call cannot be confidently cleared, it is blocked and surfaced to you for explicit confirmation (HIPS). Critical-tier protection cannot be switched off in any mode. 会 fail-open（出错即放行）的安全控制是表演。Sieve 按 fail-closed 构建：若一次 Critical 工具调用无法被确信地放行，就拦下并交给你显式确认（HIPS）。Critical 级防护在任何模式下都不可关闭。

  • Irreversible actions (signing, transfers, deploys) force a moment of cognitive friction before they can run.不可逆操作（签名、转账、部署）执行前强制制造一次"认知摩擦"。
  • If the rule engine errors or a rule is ambiguous, the call is held — not silently allowed.若规则引擎报错或规则有歧义，调用会被挂起——而不是悄悄放行。
  • Quality baseline: 0.00% Critical false-positive rate, 99.71% attack recall, with real attack-replay samples in the test suite.质量基线：Critical 误报率 0.00%，攻击召回率 99.71%，测试集含真实攻击复现样本。
Verifiable, not just trusted自证清白，而非要你盲信

Don't take our word for it — check it. 别只听我们说——你可以亲自核验。

The repository is public precisely so the claims on this page can be audited end-to-end. Four mechanisms make "trust me" unnecessary: 仓库公开，正是为了让本页的每条声明都能被端到端审计。四项机制让"相信我"变得多余：

Sigstore signaturesSigstore 签名

Every release artifact is signed and logged in a public transparency log. Verify with cosign before you install — required, not optional. 每个发布产物都经签名并记入公开透明日志。安装前用 cosign 验证——必做，非可选。

Reproducible build可复现构建

Build from source and the bytes match the published binary. The artifact you run is provably the code you read. 从源码构建得到与发布二进制一致的字节。你运行的产物，可证明就是你读过的代码。

Pinned dependencies依赖锁定

Every dependency is pinned to an exact, hash-locked version. No silent supply-chain drift between the audit and the build you ship. 每个依赖都锁定到精确、带哈希校验的版本。审计与你发布的构建之间，没有无声的供应链漂移。

Transparent rule log透明规则日志

Every rule-set update is published with a public changelog. You can see exactly what changed in your protection, and when. 每次规则集更新都附公开变更日志。你能精确看到防护规则改了什么、何时改的。
