A local security proxy between your AI agent and the model 夹在 AI agent 与模型之间的本地安全代理

Sieve is a fully local LLM-traffic security proxy — a single Rust binary that sits between your AI coding agent (Claude Code / Codex CLI / Cursor) and the upstream model API (Anthropic / OpenAI / relays). It inspects traffic in both directions — redacting secrets on the way out, and blocking dangerous tool calls on the way in (fail-closed) — to force a moment of cognitive friction before irreversible actions: signing, transfers, deploys. Built for crypto-native developers. Sieve 是一个完全本地的 LLM 流量安全代理——一个 Rust 单二进制，夹在你的 AI 编码 agent（Claude Code / Codex CLI / Cursor）与上游模型 API（Anthropic / OpenAI / 中转站）之间。它双向检查流量——出站脱敏密钥，入站阻断危险工具调用（fail-closed）——在不可逆操作（签名、转账、部署）发生前，强制注入一次认知摩擦。为 crypto-native 开发者而建。

How it works工作原理

One proxy on loopback, watching both directions回环地址上的一个代理，盯住双向流量

You point your agent's base URL at Sieve on 127.0.0.1 instead of the upstream. Sieve forwards everything to Anthropic / OpenAI / your relay and streams the response straight back — redacting secrets outbound, and intercepting Critical tool calls inbound before they reach your agent. 你把 agent 的 base URL 指向 127.0.0.1 上的 Sieve，而不是上游。Sieve 把一切转发给 Anthropic / OpenAI / 你的中转站，再把响应原样流式返回——出站脱敏密钥，入站在 Critical 工具调用抵达 agent 之前将其拦截。

Why Sieve为什么是 Sieve

Four things only this layer can do只有这一层能做的四件事

Wallet-security tools never see the prompt; DLP doesn't live in your dev workflow; existing AI-agent guards skip crypto entirely. Sieve sits in the one place that sees the whole conversation — and acts on it. 钱包安全产品看不到 prompt；DLP 不在你的开发工作流里；现有 AI agent 防护工具完全跳过 crypto。Sieve 站在唯一能看到整段对话的位置——并据此行动。

01

Exclusive LLM-traffic vantage point独占的 LLM 流量层站位

Sieve sits on the LLM traffic layer — the only spot that sees both the prompt and the model's tool calls. Wallet-security products never see the prompt; classic DLP isn't in your dev workflow at all. Sieve 处在 LLM 流量层——唯一同时看得到 prompt 与模型工具调用的位置。钱包安全产品看不到 prompt，传统 DLP 根本不在你的开发工作流里。

02

Local inference, bounded update channel本地推理 + 边界明确的更新通道

All detection runs on your machine — zero cloud dependency for any decision. The only network traffic Sieve initiates is rule and version updates, on a clearly bounded channel you can disable. 检测全部在你的机器上运行——任何判定都零云依赖。Sieve 唯一主动发起的网络流量是规则与版本更新，走一条边界清晰、可关闭的通道。

03

Crypto-specific detectionCrypto 专项检测

Built for crypto-native risks: private keys, mnemonics, signing, transfers, deploys. Across 19 surveyed LLM/DLP products and 9 AI-agent security tools, none ship crypto-specific detection. Sieve does. 为 crypto 原生风险而建：私钥、助记词、签名、转账、部署。在调研的 19 家 LLM/DLP 产品与 9 家 AI Agent 安全工具中，无一提供 crypto 专项检测。Sieve 提供。

04

Bidirectional, fail-closed双向检测 + fail-closed

Inspection runs both ways — outbound redaction and inbound interception. When detection is uncertain it fails closed, not open. Critical-tier protection cannot be turned off in any mode. 双向检查——出站脱敏与入站拦截。检测不确定时是 fail-closed，而非 fail-open。Critical 级防护在任何模式下都不可关闭。

Privacy隐私

It never uploads what you actually do它永不上传你真正在做的事

Detection is 100% local, so Sieve simply never needs your data to leave. The only outbound it initiates is an update check — four times a day — carrying just five fields, none of which identify you. 检测 100% 本地，所以 Sieve 根本不需要你的数据离开机器。它唯一主动发起的出站是更新检查——每天 4 次——仅携带 5 个字段，没有一个能识别你。

  • 1version — the Sieve build you runversion — 你运行的 Sieve 版本
  • 2OS — e.g. macOSOS — 操作系统，如 macOS
  • 3CPU arch — arm64 / x86_64CPU arch — CPU 架构 arm64 / x86_64
  • 4install-id — a random local UUID, not tied to any account or deviceinstall-id — 本地随机 UUID，不绑定账号或设备
  • 5channel — stable / betachannel — 更新通道 stable / beta
  • Never uploads your prompts, responses, API keys, or usage records永不上传你的 prompt、response、API key 或使用记录
  • Update check runs 4×/day — 5 fields only, no payload of yours更新检查每天 4 次——仅 5 字段，不含你的任何内容
SIEVE_NO_TELEMETRY=1 Stops the install-count beacon. Rule updates still work. 停止装机统计信标。规则更新照常工作。
SIEVE_NO_UPDATE=1 Disables update checks entirely. No outbound to the update server at all. 完全禁用更新检查。彻底不再连接更新服务器。

Verifiable, not just trusted可验证，而非要你盲信

Self-custody trust — check it yourself自托管信任——你可以亲自核验

A security tool that sits in your most sensitive spot must let you verify, not just trust. Four mechanisms make "trust me" unnecessary. 站在你最敏感位置的安全工具，必须让你能验证，而非仅凭信任。四项机制让"相信我"变得多余。

Sigstore signaturesSigstore 签名

Every release artifact is signed and logged in a public transparency log. Verify with cosign before you install.每个发布产物都经签名并记入公开透明日志。安装前用 cosign 验证。

Reproducible build可复现构建

Build from source and the bytes match the published binary. What you run is provably the code you read.从源码构建得到与发布二进制一致的字节。你运行的，可证明就是你读过的代码。

Pinned dependencies依赖锁定

Every dependency is pinned to an exact, hash-locked version. No silent supply-chain drift between the audit and the build.每个依赖锁定到精确、带哈希校验的版本。审计与构建之间无无声漂移。

Transparent rule log透明规则日志

Every rule-set update ships with a public changelog. You see exactly what changed in your protection, and when.每次规则集更新都附公开变更日志。你能精确看到防护改了什么、何时改的。
