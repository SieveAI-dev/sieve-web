How it works工作原理

One local proxy between your agent and the model 一个夹在 agent 与模型之间的本地代理

Sieve is a single Rust binary that runs on 127.0.0.1 and speaks the same API your agent already speaks. It inspects traffic in both directions — redacting secrets on the way out, and intercepting dangerous tool calls on the way in (fail-closed) — without you changing how you work. Sieve 是一个运行在 127.0.0.1 上的单一 Rust 二进制，说着你 agent 本就在说的同一套 API。它双向检查流量——出站脱敏密钥、入站拦截危险工具调用（fail-closed）——而你的工作方式无需任何改变。

127.0.0.1 reverse proxy · detection 100% local · fail-closed

01 — Where it sits01 — 部署位置

A reverse proxy on loopback回环地址上的反向代理

You point your agent's base URL at Sieve instead of the upstream. Sieve listens on 127.0.0.1, forwards everything to Anthropic / OpenAI / your relay, and streams the response straight back. Nothing leaves your machine except the same request that would have gone out anyway. 你把 agent 的 base URL 指向 Sieve 而不是上游。Sieve 监听 127.0.0.1，把一切转发给 Anthropic / OpenAI / 你的中转站，再把响应原样流式返回。除了本就要发出的那个请求，没有任何东西离开你的机器。

02 — Outbound02 — 出站

Redaction that never interrupts you从不打断你的脱敏

On every request leaving for the model, Sieve runs OUT-* rules — high-frequency patterns for private keys, mnemonics, API tokens, and other secrets. Matches are redacted in place before the request is forwarded. This is the common case, so it is designed to be invisible: there is no prompt, no blocking, no confirmation. 对每个发往模型的请求，Sieve 运行 OUT-* 规则——针对私钥、助记词、API token 等密钥的高频模式。命中后在转发前就地脱敏。这是最常见的情况，因此被设计成无感的：不弹窗、不阻断、不需要确认。

You only learn it happened from a passive signal: a status-bar notification that auto-dismisses after ~5 seconds. Your workflow is never paused; the secret simply never reaches the upstream. 你只通过一个被动信号知道发生了什么：一条约 5 秒后自动消失的状态栏通知。你的工作流从不暂停，密钥只是从未抵达上游。

outbound · redacted in place 
# request as your agent sent it
deploy with key 0xAC0974BEC...f80

# what actually leaves your machine
deploy with key <REDACTED:OUT-EVM-PK>

# status bar (5s, auto-dismiss)
✓ Sieve redacted 1 secret · OUT-EVM-PK
OUT-* high-frequency outbound rules — auto-redact, no friction 高频出站规则——自动脱敏、零打断

03 — Inbound03 — 入站

Critical tool calls are intercepted, fail-closedCritical 工具调用被拦截，fail-closed

The dangerous direction is the model's reply. A response can carry a tool call that signs a transaction, moves funds, or deploys. Sieve inspects every inbound tool call and, when it is Critical, holds it and forces a human decision. 危险的方向是模型的回复。一条响应可能携带一个签名交易、转移资金或部署的工具调用。Sieve 检查每一个入站工具调用，当它是 Critical 时，扣住它并强制一次人工决策。

L1 + L2

Two-layer defense双层防御

A fast pattern layer catches known-dangerous tool calls; a semantic layer reasons about intent for the cases patterns miss. Both run locally. 快速模式层捕获已知危险的工具调用；语义层对模式漏掉的情况做意图推理。两层都在本地运行。

FAIL-CLOSED

Default to block默认阻断

If detection is uncertain or errors out, the call does not slip through — it is held and surfaced for confirmation. Critical can't be turned off in any mode. 若检测不确定或出错，调用不会漏过——它被扣住并交付确认。Critical 在任何模式下都不可关闭。

HIPS

Human confirmation人工确认

For irreversible actions — signing, transfers, deploys — Sieve injects a moment of cognitive friction. You explicitly allow once or block. Nothing executes silently. 对不可逆操作——签名、转账、部署——Sieve 注入一个认知摩擦的瞬间。你显式地放行一次或阻断。没有任何东西会静默执行。

04 — Detection04 — 检测

Detection reasoning is 100% local检测推理 100% 本地

Both the outbound redaction and the inbound classification run entirely on your machine. No prompt, no response, no API key, and no usage record is ever sent anywhere for analysis. The only thing Sieve talks to the network for is rule and version updates — on a clearly bounded channel you can disable. 出站脱敏与入站分类都完全在你的机器上运行。没有任何 prompt、响应、API key 或使用记录会被发往任何地方做分析。Sieve 唯一为之联网的是规则与版本更新——一条边界清晰、可关闭的通道。

  • No prompt / response / API key ever uploaded从不上传 prompt / 响应 / API key
  • No usage records leave the machine没有使用记录离开机器
  • Updates only: 5 fields, on a bounded channel仅更新：5 个字段，边界清晰的通道
  • SIEVE_NO_UPDATE=1 disables update checks entirelySIEVE_NO_UPDATE=1 完全禁用更新检查

05 — Topology05 — 拓扑

Many listeners, many upstreams多 listener，多上游

One Sieve instance can expose several local listeners at once — one per agent or per protocol — and route each to its own upstream. Run Claude Code against Anthropic and Codex CLI against OpenAI through the same proxy, or send everything through a relay. Every path is inspected the same way. 一个 Sieve 实例可同时暴露多个本地 listener——每个 agent 或每种协议一个——并把各自路由到自己的上游。让 Claude Code 走 Anthropic、Codex CLI 走 OpenAI，都经同一个代理；或把一切发往中转站。每条路径都以同样方式被检查。

Per-agent listeners按 agent 划分 listener

Each agent points at its own local port. Add or remove agents without touching the others.每个 agent 指向自己的本地端口。增删 agent 互不影响。

Pluggable upstreams可插拔上游

Anthropic, OpenAI, or a relay — chosen per route. Switch the destination without changing your agent.Anthropic、OpenAI 或中转站——按路由选择。换目的地无需改 agent。

Uniform inspection统一检查

Outbound redaction and inbound interception apply to every route — no path is a blind spot.出站脱敏与入站拦截对每条路由都生效——没有任何路径是盲区。

Put it in front of your agent把它放到你的 agent 前面

Point your base URL at Sieve and keep working. Outbound secrets get redacted, Critical tool calls get held — without changing your workflow. 把 base URL 指向 Sieve，照常工作。出站密钥被脱敏，Critical 工具调用被扣住——而你的工作流毫无改变。
